Medium Card Specifications

  • Required Elements: Icon + object type label, Process Name, Status badge, Start Date, Last Updated, Participant summary (1-3 users), Primary action

  • Behavior: Grouped metadata, scannable format, limited actions

Large Card Specifications

  • Required Elements: Icon + object type label, Process Name (prominent), Status badge, Process summary (clamped), Metadata grid (Start Date, Last Updated, Current State, Initiating User), Expanded participant section, Related object counts, Action bar with multiple actions
  • Layout: Structured card with clear visual sections, strong hierarchy separating status, metadata, and actions
  • Behavior: Summary text truncated, related objects as counts/icons, permission-aware actions, non-destructive operations
  • Excluded: No workflow diagrams, no timelines, no BPMN visualizations

Cross-Variant Consistency Rules

Foundational principles ensuring visual and behavioral coherence across all Business Process card variants.

Visual Alignment

Business Process cards must align visually with Task and Case-related cards, maintaining consistent spacing, typography, and color schemes across the entire ArkCase design system.

Status Consistency

Status placement and styling should be consistent across all card sizes. Status badges must always appear in the top-right corner with standardized colors and typography.

Gradual Complexity Scaling

Complexity should scale gradually from Small → Medium → Large variants. Each size adds appropriate detail without overwhelming the user or breaking visual hierarchy.

Unified Component Identity

All variants should feel like scaled versions of the same component. Users should immediately recognize a Business Process card regardless of its size or context.

Design Constraints & Boundaries

Critical limitations and scope boundaries that define what Business Process cards should and should not include.

Prohibited Elements

  • Do not design BPMN diagrams or task flow visualizations
  • Do not invent workflow states or metrics not provided by the system
  • Do not assume administrative workflow access or permissions
  • Do not include workflow editing or modification capabilities

Design Focus Areas

  • Focus strictly on status, context, and presentation
  • Prioritize clear communication of process state and progress
  • Emphasize participant visibility and accountability
  • Provide quick access to related objects and context

Process Status Reference Guide

Standardized status indicators and their visual representations across all Business Process card variants.

Active

Active Process

Process is currently running with tasks in progress. Participants are actively working on assigned activities.

Color: Green (#10B981)
Completed

Completed Process

Process has successfully finished all required tasks and reached its final state. No further action required.

Color: Blue (#3B82F6)
Suspended

Suspended Process

Process is temporarily paused pending external input, approval, or resolution of blocking issues. Can be resumed.

Color: Yellow (#F59E0B)
Terminated

Terminated Process

Process was manually stopped or automatically terminated due to errors, policy violations, or administrative action.

Color: Red (#EF4444)

Interaction Patterns & Behaviors

Standardized interaction models and behavioral expectations for Business Process cards across different contexts.

Hover States

All clickable cards should provide visual feedback on hover including subtle border color changes and shadow elevation. Non-interactive cards (Large variant with action buttons) should not change on hover.

Click Behavior

Small and Medium cards: entire card is clickable and navigates to process detail view. Large cards: provide explicit action buttons for navigation and related operations.

Accessibility Requirements

All cards must include proper ARIA labels, semantic HTML, sufficient color contrast (WCAG AA minimum), and screen reader compatible content structure.

  • ARIA labels for all interactive elements
  • Minimum 4.5:1 contrast ratio for text
  • Status communicated via text, not color alone

Adaptive layout strategies for Business Process cards across different viewport sizes and device contexts.

Responsive Design Note

While these wireframes are optimized for desktop contexts, the underlying component architecture should support responsive adaptation. Touch targets should be minimum 44x44px on mobile devices, and text should remain legible at all viewport sizes.

Permission-Aware Rendering

  • Check user permissions before rendering action buttons
  • Redact sensitive participant information based on access level
  • Hide related object counts if user lacks view permissions
  • Disable navigation if user cannot access process details
  • Show restricted indicators when process data is partially hidden

State Management

  • Implement optimistic UI updates for status changes
  • Cache process data to reduce API calls on re-renders
  • Subscribe to real-time updates for active processes
  • Handle loading and error states gracefully

Performance Optimization

  • Lazy load participant avatars and related object counts
  • Use virtual scrolling for long lists of process cards
  • Implement pagination or infinite scroll for large datasets
  • Debounce hover effects to prevent excessive re-renders
