Queue Cards

Workflow queues for work item routing

These cards represent workflow queues that route, prioritize, and organize work items in ArkCase. Each variant shows queue depth, assignment rules, and item counts optimized for workflow management.

  • Optimized for quick identification in dense lists
  • Minimal vertical height for space efficiency
  • Entire card is clickable/selectable

Critical Constraint

Do not show queue rules, assignment logic, or routing conditions. This is not a configuration interface. Focus strictly on workload visibility.

Permission-Aware

All actions must respect user permissions. Some metadata or actions may be hidden, disabled, or redacted based on access rights.

Cross-Variant Consistency Rules

Visual Alignment

Queue cards must align visually with Group and Business Process cards to maintain system-wide consistency.

Naming & Iconography

Queue naming conventions and icon usage must remain consistent across all card sizes and contexts.

Progressive Disclosure

Workload indicators scale gradually from Small → Medium → Large, revealing more detail at each level.

Scaling Principle

All variants should feel like scaled versions of the same component, not different designs. Information density increases predictably, but the core identity remains constant.

Usage Guidelines

When to Use Small Cards

  • → Queue selection dropdowns or pickers in forms
  • → Filter panels where users need to select multiple queues
  • → Dense sidebar navigation lists
  • → Autocomplete search results

When to Use Medium Cards

  • → Dashboard queue overview grids (2-3 columns)
  • → Queue management list views
  • → Team workload distribution displays
  • → Queue comparison interfaces

When to Use Large Cards

  • → Inspector panels showing queue details
  • → Queue detail modal dialogs
  • → Expanded preview states in split-view layouts
  • → Standalone queue overview pages

Design Constraints & Boundaries

Do Not Include

Queue Configuration UI
No rule editors, routing logic, or assignment configuration interfaces
Individual Work Items
Cards show workload summaries only, never lists of specific cases or tasks
Administrative Settings
No permission management, user assignment, or system configuration options
Complex Workflows
Avoid showing routing conditions, business rules, or automation triggers

Must Include

Clear Object Type
Always identify the card as representing a "Queue" object
Workload Context
Show item counts and status breakdowns appropriate to card size
Purpose Description
Explain what type of work the queue handles and how it's used
Navigation Actions
Provide clear paths to view queue contents or access related functions

Interaction Patterns

Click Behavior

Small: Entire card clickable
Medium: Card + primary button
Large: Multiple action targets

Hover States

Border color change
Subtle background shift
Optional shadow elevation

Keyboard Nav

Tab focus on card/actions
Enter/Space to activate
Arrow keys in lists

Accessibility Requirements

Screen Reader Support

  • All icons paired with descriptive text or aria-labels
  • Semantic HTML structure (heading hierarchy)
  • Status indicators announced with role="status"
  • Button labels clearly describe action outcome

Visual Considerations

  • Minimum 4.5:1 contrast ratio for text
  • Color never sole indicator of status
  • Focus indicators clearly visible (2px outline)
  • Click targets minimum 44x44px touch area

Implementation Notes

Component Architecture

  • Build as a single component with size prop (small | medium | large)
  • Use conditional rendering based on variant to show/hide elements
  • Accept queue data object with all possible fields
  • Handle missing optional data gracefully

Data Requirements

  • Queue ID, name, and type (required)
  • Description/purpose text
  • Workload counts (total, by status)
  • Metadata (created date, display order, etc.)
  • Optional: Priority level, SLA status

Responsive Behavior

  • Small cards remain consistent across breakpoints
  • Medium cards may stack in single column on mobile
  • Large cards adapt to full-width on smaller screens
  • Consider collapsing metadata grid to single column

Performance Tips

  • Virtualize lists when rendering 50+ small cards
  • Lazy load workload data for large card expansions
  • Cache queue metadata to reduce API calls
  • Use skeleton loaders during data fetch

Accessibility

Keyboard navigation and screen reader support for Queue cards.

Keyboard Navigation

  • Tab: Navigate between queue cards
  • Enter/Space: Open queue or claim item
  • Arrow Keys: Navigate within queue items
  • Escape: Close dialogs or deselect

Screen Reader Support

  • Semantic HTML with proper list structure
  • ARIA labels for queue status and workload
  • Item counts announced dynamically
  • Priority levels clearly indicated
