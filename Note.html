Design Guidelines

Core principles governing Note card design across all variants

Authorship Clarity

Author identity must be immediately clear

  • Always display author name prominently
  • Use avatars for human authors in Medium/Large cards
  • Distinguish system-generated notes with appropriate iconography
  • Maintain consistent author placement across all variants

Temporal Context

Time information must be contextually appropriate

  • Use relative time ("2h ago") for Small cards in recent activity
  • Show full timestamps in Medium cards for clarity
  • Include both created and modified dates in Large cards
  • Ensure timestamps are always visible and legible

Visual Hierarchy

Content priority must guide visual weight

  • Note content is primary; metadata is secondary
  • Use typography scale to establish clear hierarchy
  • Group related information visually
  • Maintain visual balance across card variants

Access Control

Visibility and permissions must be clear

  • Clearly distinguish Internal vs. General notes
  • Use consistent visual indicators for restricted content
  • Show permission-aware actions only
  • Include access warnings for sensitive notes

Contextual Relationship

Parent object connection must be evident

  • Show parent context in Large cards with clear linking
  • Keep parent reference visually subordinate to note content
  • Use consistent parent object card styling
  • Provide clear navigation to parent object

Progressive Disclosure

Information scales with card size

  • Small cards show minimal essential information
  • Medium cards add context and preview
  • Large cards reveal full content and relationships
  • Each variant feels like natural progression

Implementation Note

Note type badges should use consistent color coding across the entire system. General notes use blue, Internal notes use red, and System notes use purple. This color system helps users quickly identify note types at a glance, especially in dense lists and activity feeds.

Responsive Behavior

How Note cards adapt to different contexts and container sizes

Narrow Containers

When displayed in narrow side panels or mobile viewports:

  • Small cards maintain single-line layout
  • Medium cards stack metadata vertically
  • Large cards reduce horizontal padding
  • Action buttons may stack or collapse to menu

Grid Layouts

When displayed in grid or multi-column layouts:

  • Cards maintain consistent heights within rows
  • Content clamps at defined line limits
  • Metadata truncates with ellipsis if needed
  • Hover states provide expansion affordance

List Views

When displayed in vertical list layouts:

  • Cards span full container width
  • Consistent spacing between cards
  • Alternating backgrounds optional for density
  • Scroll performance optimized with virtualization

Inspector Panels

When displayed in inspector or detail panels:

  • Large cards preferred for maximum context
  • Full content display without truncation
  • Related notes may appear as Medium cards
  • Scrollable container for multiple notes

Interaction Patterns

Visual feedback for user interactions across all card variants

Default

Resting state with standard styling

  • • Border: gray-200
  • • Background: white
  • • Shadow: none
  • • Cursor: default or pointer

Hover

Indicates interactivity and clickability

  • • Border: gray-300
  • • Background: white
  • • Shadow: subtle elevation
  • • Transition: smooth 200ms

Focus

Keyboard navigation and accessibility

  • • Border: primary-500
  • • Background: white
  • • Outline: 2px primary ring
  • • Keyboard accessible

Active/Selected

Currently selected or active note

  • • Border: primary-500
  • • Background: primary-50
  • • Shadow: medium elevation
  • • Persistent until deselected

Accessibility Considerations

All interactive states must be keyboard accessible and provide clear visual feedback. Focus states should be distinct from hover states, and all clickable elements must have appropriate ARIA labels and roles for screen readers.
