Design Principles & Guidelines

Core principles governing subscription card design in ArkCase

Clear Object Identity

Every subscription card must immediately communicate what object is being subscribed to, with clear visual hierarchy distinguishing the subscription from the subscribed object.

Implementation
  • • Subscription icon always visible
  • • Object type clearly labeled
  • • Primary title uses consistent formatting

Subscriber Prominence

The subscribed user must be immediately identifiable through avatar, name, or both depending on card size, ensuring clarity about who will receive notifications.

Implementation
  • • Avatar in all variants
  • • Name visible in medium and large
  • • Role/title in large cards

Scalable Context

Information density must scale appropriately from small to large cards, with each variant feeling like a natural expansion of the previous, not a completely different design.

Implementation
  • • Consistent visual language
  • • Progressive information disclosure
  • • Predictable layout expansion

Permission Awareness

Subscription cards must respect permission boundaries, showing or hiding actions like unsubscribe based on user roles and object access rights.

Implementation
  • • Conditional action visibility
  • • Disabled states for restricted actions
  • • Clear permission indicators

No Notification Mechanics

Subscription cards focus on subscription context, not notification delivery. Avoid showing notification types, schedules, or delivery preferences.

What to Avoid
  • • Notification frequency settings
  • • Delivery channel preferences
  • • Message templates or rules

Status Visibility

Subscription state (active, paused, expired) must be clearly visible and consistently styled across all card variants to prevent confusion.

Implementation
  • • Color-coded status badges
  • • Consistent badge positioning
  • • Explanatory text in large cards

Behavioral Specifications

Interaction patterns and behavioral requirements for subscription cards

Small Card Behavior

1
Entire Card Clickable
The entire card acts as a single click target, navigating to the subscribed object or opening a detailed view.
2
No Visible Actions
Actions like unsubscribe or settings are not shown to maintain minimal height and visual simplicity.
3
Hover State
Subtle background color change and border emphasis on hover to indicate interactivity.
4
Text Truncation
Long object names are truncated with ellipsis to maintain single-row or tight column layout.

Medium Card Behavior

1
Primary Action Visible
A single primary action button (e.g., "View Case") is always visible at the bottom of the card.
2
Secondary Actions Menu
Additional actions accessible through a more menu (three-dot icon) to avoid cluttering the interface.
3
Status Badge Interaction
Status badges are non-interactive but may show tooltips on hover explaining the current state.
4
Subscriber Click
Clicking on subscriber avatar or name opens the user profile or contact card.

Large Card Behavior

1
Multiple Actions
All relevant actions (View, Unsubscribe, Pause, etc.) are visible in an action bar at the bottom of the card.
2
Permission-Aware Unsubscribe
Unsubscribe button is only enabled if the current user has permission to modify the subscription.
3
Metadata Grid Interaction
Metadata fields may be clickable to show more details or related objects (e.g., clicking "Created By" shows user profile).
4
Object Context Card
The subscribed object details section is clickable and navigates to the full object view.
5
Status-Specific Actions
If subscription is paused, a "Resume" action appears. If expired, a "Renew" action may be shown.

Common Use Cases

Real-world scenarios showing subscription cards in context

Subscription List View

User viewing all their active subscriptions in a dedicated subscriptions panel or page. Medium cards provide the right balance of information and density.

Recommended Card Size
Medium Card (25.2)

Related Object Panel

Sidebar showing related subscriptions for a case or task. Small cards maximize space efficiency while maintaining clarity.

Recommended Card Size
Small Card (25.1)

Inspector Panel

Detailed view when user clicks on a subscription to see full context, metadata, and available actions. Large cards provide comprehensive information.

Recommended Card Size
Large Card (25.3)

Dashboard Widget

Dashboard showing recent subscriptions or subscription activity. Medium or small cards depending on widget size and information needs.

Recommended Card Size
Medium Card (25.2) or Small Card (25.1)

Accessibility Considerations

Ensuring subscription cards are usable by all users

Keyboard Navigation

All interactive elements must be keyboard accessible with visible focus indicators. Tab order should follow visual hierarchy.

Screen Reader Support

Proper ARIA labels for icons, buttons, and status indicators. Announce subscription type, object, and subscriber information.

Color Contrast

Status badges and text must meet WCAG AA standards (4.5:1 for normal text, 3:1 for large text). Never rely on color alone.

Text Sizing

Support browser text zoom up to 200% without loss of functionality or content overlap. Use relative units for sizing.

Error States

Clear error messaging for failed actions with specific instructions. Errors announced to screen readers via aria-live regions.

Touch Targets

Minimum 44x44px touch target size for all interactive elements. Adequate spacing between clickable areas to prevent mis-taps.
