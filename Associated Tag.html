Interaction Patterns

Interaction patterns and behavioral expectations for Associated Tag cards across all variants.

Small Card Behavior

  • Entire card is clickable/selectable
  • No visible actions or menus
  • Hover state provides visual feedback
  • Click navigates to parent object
  • Visual density matches tag small cards

Medium Card Behavior

  • Parent reference visually secondary
  • Primary action clearly labeled
  • Secondary actions in overflow menu
  • Avoids repeating tag definition details
  • Minimal and predictable actions

Large Card Behavior

  • Attachment context dominates display
  • Multiple actions available in action bar
  • Actions are permission-aware
  • Non-destructive actions by default
  • Does not show system-wide tag usage

Design Guidelines

Critical limitations and scope boundaries for Associated Tag card implementations.

Tag Creation & Taxonomy

Do not design tag creation or taxonomy management interfaces within these cards.

Rationale: These cards represent tag attachments, not tag definitions. Creation and management belong in dedicated administrative interfaces.

Internal IDs & Rules

Do not expose internal IDs, database keys, or tagging rules to end users.

Rationale: Technical implementation details create confusion and have no user value. Display only human-readable identifiers.

Permission Assumptions

Do not assume tag removal permissions or destructive action capabilities.

Rationale: Permission models vary by implementation. All destructive actions must be permission-aware and require confirmation.

Focus on Attachment Context

Focus strictly on attachment context and classification, not tag usage analytics.

Rationale: These cards show one specific attachment. System-wide usage belongs in analytics dashboards and reporting tools.

Accessibility

Keyboard navigation and screen reader support for Associated Tag cards.

Keyboard Navigation

  • Tab: Navigate between tag cards
  • Enter/Space: Select tag or navigate to parent
  • Delete/Backspace: Remove tag association (with confirmation)
  • Escape: Cancel tag operations

Screen Reader Support

  • Tag name and color category announced
  • Parent object context clearly labeled
  • Association date and creator announced
  • ARIA labels for remove/edit actions

Implementation Notes

Technical considerations for Associated Tag card development.

Component Architecture

  • Compose from base Tag card with parent context
  • Support dynamic color theming from tag definition
  • Handle bidirectional navigation (tag ↔ parent)

Performance

  • Cache tag color mappings for consistency
  • Lazy load parent object details on hover/expand
  • Batch tag operations for bulk updates
