User Object — Card Variants

Enterprise card components for representing system users in ArkCase

Design Specifications

These wireframes represent User objects (internal staff accounts) across three density levels: Small (compact lists), Medium (default views), and Large (inspector panels). All variants maintain visual consistency with Person and Organization cards.

User — Small Card

Compact representation for participant lists, assignment pickers, and dense views. Optimized for quick identification with minimal vertical height.

Design Consistency Rules

Guidelines for maintaining visual harmony across all User card variants

Visual Alignment

User cards must align visually with Person and Organization cards. Avatar placement, name hierarchy, and status indicators should follow identical patterns across all object types.

Progressive Disclosure

System metadata should scale gradually from Small to Large variants. Each size should feel like a natural expansion of the previous, not a completely different component.

Status Indicators

Active status uses green badges, Disabled uses gray. Restriction indicators appear as amber badges with lock icons. Status must be immediately visible in all variants.

Privacy & Security

Never expose sensitive directory attributes (DNs, internal IDs, passwords). Group memberships shown as counts only. Actions must respect permission boundaries.

Typography Hierarchy

Display Name is always the primary text element. Secondary text (role, department) uses smaller size and muted color. System metadata uses the smallest text size.

Interactive Behavior

Small cards are fully clickable. Medium cards have a single action button. Large cards provide multiple actions with overflow menu. Hover states must be consistent.

Implementation Notes

Technical considerations for developers implementing User cards

Component Architecture

  • Use a single base component with size prop (small/medium/large)
  • Implement conditional rendering based on variant
  • Share styles and utilities across all card types
  • Support both avatar images and initials fallback

Data Requirements

  • displayName, accountStatus (required)
  • role, department, organization (recommended)
  • associations: caseCount, taskCount, workflowCount
  • groupMembershipCount, lastLogin, accountCreated

Security Considerations

  • Never expose LDAP DNs or internal user IDs
  • Respect permission boundaries for actions
  • Filter email visibility based on context
  • Show group memberships as counts only

Accessibility

  • Include ARIA labels for all interactive elements
  • Ensure keyboard navigation support
  • Provide text alternatives for status icons
  • Maintain sufficient color contrast ratios
